项目概览

PhysiBuilder 研究的是 text-to-3D scene generation 的一个关键问题：自然语言能否不只生成一个视觉上合理的物体，而是生成一个多物体、物理关系合理、后续可编辑的 3D 场景。现有 text-to-3D 方法在单物体上进展很快，但多物体场景容易出现 Janus problem、物体关系混乱、随机位置不稳定和生成后难以编辑。

我作为一作推进 2024 暑期 Paid RA 工作，设计 ScenePlannerLLM + SceneConstructor。核心思想是先让 LLM 生成符合物理常识的 3D scene prior，再由 SceneConstructor 用 instance-specific NeRF 建模每个对象。编辑时改布局和对象状态，而不是重新训练整个场景。

1. 问题： 3D scene 不能只靠最终图像判断

多物体 3D 生成有两个层面的困难。第一是物体关系：桌子、椅子、书、杯子之间需要有位置、支撑、碰撞和语义关系，不能随机漂浮或互相穿透。第二是可编辑性：用户生成场景后往往想移动、复制、删除某个对象，而不是重新输入 prompt 等系统再采样一次。

这让 Physibuilder 走向一个中间表示：不要让 diffusion 或 NeRF 直接承担全部推理，而是先产生 3D layout / scene graph / physical prior。这个中间产物可以检查、修改、重用，也让生成链路不再是黑箱。

2. 方法： ScenePlannerLLM + SceneConstructor

ScenePlannerLLM 把文本 prompt 转成 3D layout 和 scene graph，使用 chain-of-thought / self-consistency，并用 gravity / collision 检查修正布局。它输出的不只是文字解释，而是生成模块可以消费的 scene prior。

SceneConstructor 为每个对象建立 instance-specific NeRF，从 local coordinates 映射到 scene coordinates。这样每个对象可以独立生成和编辑，场景组合保持可控。移动、复制、删除等交互操作通过改变对象级布局完成，不需要重新 fine-tuning 整个场景。

3. 实验

论文对比 DreamFusion、HiFA、Magic3D、ProlificDreamer 等 text-to-3D 方法。Scene generation user study 中，Ours 在 scene quality 7.8、text alignment 7.3、identity consistency 8.5 上达到最优或接近最优。Layout physical metric 中，Ours gravity 0.88、collision 0.96，高于 Standard / CoT / Ours w/o physics。

Layout user study 中，Ours physical 7.8、scene quality 7.8、scene composition 8.0。更重要的是编辑能力：由于对象以 instance-specific 方式组织，系统可以支持 move、duplicate、remove 等自然语言交互式 3D 编辑。

4. 这段研究训练了什么

PhysiBuilder 对我后续做 agent workspace 很有影响。它训练的是“先把模型能力变成可检查的中间产物”的习惯。在 3D 里，这个中间产物是 layout 和 scene prior；在 Lawless 里，它变成 case facts、source list、claim matrix、review queue 和 artifact trace。

这类系统的共同点是：不要把所有智能都压进最终 response。让模型先计划、把计划落成结构，再用验证器检查结构，最后生成用户看到的结果。

5. 边界

PhysiBuilder 仍受 SDS 优化成本、背景模糊、非盒状对象和圆角形状表达限制。LLM layout 也不能保证所有物理关系都正确，仍需要后续几何和视觉检查。这个项目的价值不是宣称 text-to-3D 已解决，而是展示一个可编辑、多对象、带中间表示的生成框架。

和普通 text-to-3D 方法的区别

很多 text-to-3D 方法直接从 prompt 走向生成结果，问题是多物体场景的布局、支撑、碰撞和后续编辑都很难保证。PhysiBuilder 的不同点是把 LLM reasoning 放在生成前：ScenePlannerLLM 先产生可检查的 3D scene prior，再由 SceneConstructor 构建对象。

它也不是把 LLM 当作万能生成器。LLM 负责结构、关系和物理常识的规划；instance-specific NeRF 负责对象级表示和编辑。这种分工让 move、duplicate、remove 等操作成为系统能力，而不是重新采样整场景。

它也不是 2026 年唯一的 layout-guided 方向。现在 text-to-3D scene generation 已经明显走向 semantic layout、scene graph、3DGS 和 physics-guided 控制。PhysiBuilder 的价值是我在 2024 年一作推进了同一类核心判断：先产出可检查 scene prior，再让生成模块消费它。

6. 技术补充： 为什么 LLM 适合做 scene prior

LLM 不擅长直接生成 3D geometry，但擅长把自然语言里的关系拆成结构。比如“a chair next to a table with a lamp on it”包含支撑、相邻、对象类别和空间关系。ScenePlannerLLM 的价值就在于把这些隐含关系显式化，生成 layout 和 scene graph，再交给视觉生成模块。

这种中间表示让系统可以在生成前检查 gravity 和 collision。没有这一步，多物体 text-to-3D 容易把对象放在不合理位置，或者在不同视角里出现身份漂移。Scene prior 不保证最终渲染完美，但它给生成链路一个可控起点。

SceneConstructor 用 instance-specific NeRF 建模对象，也是一种解耦：对象内部的外观学习和场景级位置关系分开。编辑时移动对象，不需要重新训练对象；复制对象，也不需要从 prompt 重新采样。

7. 实验补充

用户研究和 physical metric 的作用不同。User study 评估 scene quality、text alignment 和 identity consistency，反映人类是否觉得结果符合 prompt；gravity/collision metric 则直接检查布局物理性。两者结合，才能说明系统既看起来合理，也在结构上更稳定。

ablation 里去掉 physics 或 self-consistency 会降低布局质量，这说明提升并不只来自更长 prompt，而是来自规划、检查和修正的组合。这个结论和后续 agent 工作非常相似：不是让模型一次写完，而是让它在结构化中间产物上迭代。

8. 贡献边界

PhysiBuilder 的贡献可以浓缩成三点。第一，它把 text-to-3D 从单物体扩展到多物体 scene，并把 layout 作为显式中间产物。第二，它让 LLM 的世界知识参与 3D 场景规划，但没有让 LLM 直接承担几何生成。第三，它用 instance-specific NeRF 支持对象级编辑，让用户能够移动、复制、删除对象，而不是重新生成整个场景。

这三点比单纯列 2D diffusion、NeRF、LLM 更有意义。技术名词不是主角，主角是一个自然语言请求如何被拆成可检查、可生成、可编辑的场景构建过程。

9. 补充说明： 和 agent 工作的共同结构

PhysiBuilder 和 Lawless 看似相距很远，一个是 3D vision，一个是法律/税务 agent，但它们共享同一个系统结构：先把自然语言转成可检查的中间表示，再让生成或执行模块消费这个表示，最后用验证或人类反馈检查结果。在 PhysiBuilder 里，中间表示是 layout、scene graph 和 physical prior；在 Lawless 里，中间表示是 case facts、source list、claim matrix、path cards 和 artifact trace。

这个结构比“LLM + domain tool”更稳定。LLM 的优势是把模糊意图拆成结构，domain tool 的优势是执行和验证。系统设计的任务就是让两者之间的接口清楚。PhysiBuilder 的研究经历让我更早意识到，好的 AI 系统不是让模型一次性完成所有事，而是把任务分成计划、构建、检查和编辑。

10. 补充说明： 为什么编辑能力重要

交互式编辑是 Physibuilder 区别于很多 text-to-3D demo 的地方。用户生成一个场景后，常常只想改一部分：把椅子挪到桌子旁边，复制一个杯子，删除一个不合适的对象。如果系统只能重新生成整场景，用户很难稳定迭代。对象级表示让编辑成为模型能力的一部分，而不是后处理技巧。

11. 结论

PhysiBuilder 的核心不是“LLM + NeRF”，而是把自然语言先转成可检查的 3D scene prior，再让生成模型构建对象并支持对象级编辑。这个方法贡献也能自然连接到我后来做 agent workspace 时对 intermediate artifact、verification 和 editing loop 的重视。
