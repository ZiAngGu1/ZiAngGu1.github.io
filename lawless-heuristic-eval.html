Heuristic Eval Loop: 让 agent skill 像软件系统一样被测试和改进

Heuristic Eval Loop 把法律 AI 的失败样例变成可回放、可评分、可回滚的 harness 更新。

heuristic learninggraderledgerrollbackholdout
真实任务

专利检索漏掉高风险专利或污染 bucket 时，需要知道失败来自 query、source、parser、ranking 还是 verifier。

核心做法

answer-blind runner、grader、ledger、snapshot/restore、train/holdout split 和 pairwise review。

结果边界

改进对象是 skill/search strategy/source policy；分数衡量内部任务质量，最终法律结论仍需专家复核。

1. 为什么做：法律 AI 的失败不能只靠一句新提示词修

如果 patent search agent 漏掉高风险专利，只写一句“以后更仔细搜索”没有价值。失败可能来自 query 太窄、source selection 错误、design/utility 口径混淆、parser 没抓到 claim、ranking helper 把关键结果压低，或者 verifier rubric 没惩罚弱证据。

法律 AI 的改进必须能解释、能回放、能回滚。否则每次 prompt 改动都可能修好一个训练样例，破坏另一个 heldout 场景。Heuristic Eval Loop 把失败样例变成工程材料：trace、score、candidate change、keep/discard reason 和 rollback point。

2. 怎么做：让 skill 像软件系统一样迭代

runHeuristicLoop 适合 task-level 例子：对每个 case 运行 runner，写 artifacts，跑 verifier，计算 trainScore、holdoutScore 和 generalizationGap，并把结果写入 ledger.jsonl 和 summary.md。runner 在 answer-blind 条件下不能读答案文件，先独立产出报告。

runHeuristicSystemLoop 更接近真实维护：先 snapshot workspace，再让 candidate builder 修改 skill/tool/rubric，随后跑 eval 和 pairwise review。如果候选策略提升目标指标且没有伤害 heldout，就 keep；否则 restore 到 best known workspace。rollback 是一等操作，和 keep/discard 一起进入 ledger。

ledger 记录每轮 candidate change、case result、train/holdout score、generalization gap、review decision 和 keep/discard reason。这样 agent 不会反复尝试已失败方向，也能解释为什么某条规则被保留。

3. 结果是什么：专利 skill 的提升可以被证明

在 patent search 中，answer-blind runner 先独立生成报告；grader 再检查 high-risk recall、target recall、source support 和 evidence matrix。失败进入 trace artifacts，maintainer 读 trace 后修改 search strategy、source policy、ranking helper 或 verifier rubric。

约 100 个内部标注风险样例上，top-k recall 从约 10% 提升到约 90%。这个数字来自内部 benchmark，证明 skill harness 可改进；它不证明法律结论正确，也不替代律师复核。

4. 边界：分数服务于专家复核

Heuristic eval 的分数是 system-level skill quality：是否召回关键对象、是否引用来源、是否保持 train/holdout 泛化、candidate change 是否值得保留。最终法律判断仍然属于 reviewer。这个边界让系统既能持续学习，又不会把内部指标包装成法律结论。
