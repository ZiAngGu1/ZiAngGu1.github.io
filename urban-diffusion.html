项目概览

Urban Planning Diffusion 这个名字容易误导。公开 GitHub README 显示，repo README 实际上是 Video Diffusion - Pytorch，描述为 Jonathan Ho Video Diffusion Models 的 PyTorch implementation，包含 3D U-Net、GaussianDiffusion、text conditioning、GIF training pipeline、Moving-MNIST samples 等。

这个 artifact 证明我较早接触过 diffusion / video generation 的代码结构和采样流程；当前材料不支持城市规划数据集、规划指标或原创 urban planning research 的结论。

1. 问题： 证据边界决定项目定位

这个项目最有价值的地方是边界清楚：公开 repo 能证明 video diffusion implementation，因此它承担生成式视觉背景线索，而不是完整 urban-planning research。

它更适合放在生成式视觉能力线索里，补充说明我对 diffusion 代码组织、采样和视觉生成流程有早期实践；主线项目仍然是 Lawless、1shanai、PhysiBuilder 和 ICCV ReID。

2. 方法与实现

repo 包含 video_diffusion_pytorch.py、setup.py、3d-unet.png、fireworks.webp 和 Moving-MNIST samples。技术上可写 space-time factored U-Net、GaussianDiffusion、text conditioning 和 GIF training pipeline。

它和后来的 1shanai / PhysiBuilder 有连续性：1shanai 把图像生成接到用户偏好和实物交付；PhysiBuilder 把生成模型接到 3D scene prior 和 editable NeRF。这个早期 artifact 更像技术土壤，而不是最终成果。

3. 边界

当前证据不包含 urban planning 数据、城市布局、规划评估指标或训练结果；可支撑的结论是 video diffusion / generative vision 代码备份。

和常见夸大式项目叙事的区别

旧标题容易让人以为这是完整 urban planning research，但公开 repo 更准确地支持 video diffusion / generative vision artifact。因此项目被定位为补充证据，不包含城市规划数据、规划指标或原创研究结果。

和夸大式项目叙事相比，它的价值在于证据边界清楚：这个项目作为 diffusion 代码背景是成立的，作为主线研究成果不成立。

因此它是补充证据，不是核心项目：一个早期 video diffusion code artifact，说明我接触过 3D U-Net、GaussianDiffusion 和 video sampling，但不证明城市规划研究。

4. 技术补充： 公开 repo 能支持什么

公开 repo 支持的范围是 video diffusion implementation：3D U-Net、space-time factorization、GaussianDiffusion、text conditioning 和 GIF training pipeline。城市规划指标、规划数据集或城市布局优化不在当前证据范围内。

这并不代表项目没有价值。早期生成式视觉代码让人理解 diffusion 的基本工程形态：输入如何组织，噪声如何逐步去除，采样如何产生序列，模型代码如何暴露给训练脚本。这些经验后来可以迁移到 1shanai 的图像生成和 PhysiBuilder 的生成式视觉研究。

作为补充证据，它说明我接触过 diffusion pipeline，但不会把一个公开代码备份包装成不存在的研究成果。

5. 项目定位

它承担 generative vision 的背景线索：说明我不是只在 2025 年才接触生成模型，而是在更早的代码层面就接触过 diffusion pipeline。真正有产品或研究深度的视觉项目，是 1shanai 和 PhysiBuilder。

6. 证据补充： 为什么改名更好

保留 diffusion 线索、弱化 urban planning，是因为公开证据支持前者而不支持后者。这个项目最适合的定位是 generative vision artifact：展示 diffusion 代码阅读、组织和备份能力。

如果未来找到原始 urban planning 材料，例如城市布局数据、训练脚本、评估图或报告，这个项目可以升级为完整项目。当前它是 video diffusion / generative vision 的早期公开入口，连接到后来的 1shanai 和 PhysiBuilder。

7. 技术阅读笔记

Video diffusion 的工程关键是把空间和时间同时建模。3D U-Net 不只是把图片 U-Net 换个名字，而是让卷积或注意力在 frame dimension 上处理动态信息。GaussianDiffusion 负责 forward noise 和 reverse denoising schedule，training pipeline 则把视频或 GIF 数据组织成模型可学习的张量。

这些细节说明我理解 repo 的真实技术范围，也能自然连接到后续视觉项目：当输入从视频变成服装图或 3D scene，核心问题仍然是如何组织生成链路、如何检查结果、如何把用户意图传进去。

8. 补充说明： 与主线能力的关系

Urban Diffusion 支持的是生成式视觉背景：它说明我接触过 diffusion 代码，但不承担 Lawless、PhysiBuilder 或 ICCV ReID 那样的主线证据角色。

它仍然有意义。Video diffusion 让人理解生成模型的 temporal dimension，3D U-Net 让人看到空间和时间如何一起建模，GaussianDiffusion 让人理解 noise schedule 和 reverse process。这些概念后来在更复杂项目中会以不同形式出现。

当前没有证据支持城市规划数据、规划指标或原创结果，因此用“Video Diffusion Artifact”作为标题，把旧 slug 保留为 URL 兼容，并把内容限定在证据支持的范围。

9. 补充说明： 早期 artifact 的连接方式

这个小 artifact 能连接到后来的系统经验，但连接方式要准确。Video diffusion 连接的是生成式视觉的技术背景，不是用户交付；1shanai 连接的是生成图到实物；PhysiBuilder 连接的是生成模型到 3D scene prior；Lawless 连接的是 agent 到专业服务工作流。

证据边界决定项目承担的角色：Lawless 有本地 repo、截图、测试和产品细节，所以承担主线；PhysiBuilder 和 ReID 有论文材料，所以承担研究线；Urban Diffusion 证据较薄，所以只作为公开代码 artifact 呈现。

10. 补充说明： 后续如何升级

后续升级方式不是继续加形容词，而是补证据：加入训练样例、生成结果、loss 曲线、采样脚本、模型配置、数据说明，或者找回原始 urban planning 实验。到那时它才可以从代码 artifact 升级为完整项目。

11. 结论

Urban Diffusion 的边界很明确：它不假装有完整城市规划研究，而是展示 video diffusion / generative vision 代码 artifact。这个定位让项目本身更可信，也避免弱证据被包装成强结论。
