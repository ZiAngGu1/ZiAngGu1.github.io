07 / 服装

当 AI T 恤离开屏幕

1shanai 测试的是生成式服装能不能穿过用户偏好、正反面一致性、打样、下单和实物反馈这一整条链路。

AI 服装LoRA偏好打样生产
为什么重要

AI 服装如果只停在图片 gallery，就无法回答用户是否愿意确认、等待打样并对实物负责。

怎么做

项目从 T-shirt 切入，把用户选择、正反面图、prompt/LoRA 控制、订单交接和实物反馈连成闭环。

结果 / 边界

100+ beta feedback 和约 40-50 件实物样衣支持小规模产品循环；不代表成熟供应链规模。

Core Claim

一张生成出来的 T-shirt 图像，可能在屏幕上已经很完整，但产品本身还没有完成。

真正的第一道检验通常很安静。用户从几组正反面效果里选一组，说“这个颜色可以，但背面太满”，或者要求把图案缩小一点。设计仍然停留在屏幕上，但问题已经变了：系统不再只是生成一张令人惊艳的图片，而是要记住用户偏好，保持同一件衣服的正反面一致，准备可以打样的版本，并让实物反馈影响下一轮生成。

这是 1shanai 对我最有价值的经验。它是一个早期 AI apparel loop，从 T-shirt 开始，把用户偏好、正反面 flat generation、sampling、order handoff 和 delivery 连接起来。作为 co-founder，我带过一个 7 人暑期团队，把生成概念推进到小批量实体衣服。可验证证据必须写得克制：100+ beta 用户反馈信号，约 40-50 件实体衣服经过打样和交付。这个规模足够研究产品闭环，不足以声称成熟供应链、大规模销售或完整 fashion platform。

1shanai 的核心产品判断是：AI 服装生成变得真实，不是因为图像好看，而是因为用户偏好能穿过生产流程。用户喜欢一张图是弱证据；用户在候选中选择、确认正反面、等待样衣、反馈印花和版面，再让这些反馈改变下一轮生成，才是更强的证据。

把服装生成当成图库，是第一个错误

很多 AI fashion demo 让第一张截图承担太多意义。它展示一件漂亮衣服，可能还有模特图和风格变体。图像很有情绪感染力，因为衣服本来就是视觉对象。但这也是不完整的，因为服装不是作为图片被消费的。

一件 T-shirt 至少有两重生命。第一重是视觉设计：图案、颜色、风格、主题、版面。第二重是生产对象：印花位置、大小比例、正反面一致性、样衣质量、下单状态、包装和交付。这两重之间的间隙，正是很多 AI apparel idea 变弱的地方。模型可以生成很醒目的正面图案，但背面漂移成另一套设计；它可以在正方形截图里很好看，但放到可印刷区域后构图很尴尬；它可以满足 prompt，却没有抓住用户真正的品味，因为用户并不知道如何一次性把偏好写成文字。

1shanai 从相反的压力出发。用户不是 prompt engineer。用户带来的是风格灵感、颜色、图案主题、版型预期、细节，有时甚至只是一个模糊情绪。产品必须把这些信号转成结构化生成控制，再返回可选择的候选。选择比原始 prompt 更重要，因为选择承载了被观察到的偏好。

这件事看起来小，却会改变系统。如果用户拒绝某个候选是因为“正面很好，背面太复杂”，下一轮不应该把它当成普通 dislike。系统应该保留正面语义方向，降低背面复杂度，并保持色彩和图形语言一致。如果用户选择黑色 oversized T-shirt 加抽象波纹，系统不应该因为另一张图更戏剧化，就漂移到完全无关的背面主题。

真正的产品对象是一个逐步收窄的 loop：偏好输入、候选生成、正反面一致性、用户选择、可选 refinement、打样、交付和反馈回流。

为什么从 T-shirt 开始是正确约束

T-shirt 让产品约束变得具体。

服装品类很容易迅速变复杂。全品类服装会牵涉打版、尺码、面料行为、缝制结构、版型差异、生产 MOQ 和供应商协作。这些问题都重要，但它们会让第一轮产品测试变得很难解释。如果用户不喜欢最终成品，到底是模型误解了风格，正反面不一致，尺码不合适，面料手感不对，还是生产伙伴印花偏差？

T-shirt 把歧义缩小了。基础形态熟悉，用户理解正反面，印花位置能直接检查，打样速度也比复杂服装更快。团队可以问一个更清楚的问题：生成式设计意图能不能通过用户确认，变成实体衣服，并产生有用反馈？

这个问题比更大的承诺更有价值。早期 AI 产品要找的是最短的诚实闭环，让模型输出和现实发生接触。对 1shanai 来说，闭环不是“生成一件酷衣服”，而是：

捕捉用户的 style intent。

把 intent 转成 prompt control 和结构化设计约束。

生成多组正反面候选 flats。

让用户选择并 refine 一个方向。

把确认后的设计交给打样流程。

将实体衣服和确认图对照。

把实物反馈带回下一轮生成和筛选。

这个 loop 才是产品。模型只是其中一个环节。

偏好更容易被观察，而不是被问出来

很多用户不能用一句 prompt 描述自己想要的衣服，但他们能做有意义的选择。

这对消费级生成系统很重要。prompt-first 产品默认用户能把 taste 翻译成语言。有些人可以，但更多人只会说“更未来感”“简洁一点”“streetwear”“不要太吵”“看起来高级”。这些词有用，但不够具体。缺失信号会在候选比较时出现。

在 1shanai 的 workflow 里，偏好输入包括风格灵感、颜色、图案、fit、silhouette、细节、文字或 branding。系统再把这些输入转成控制项：garment type、fit、colorway、graphics、placement、texture 和 mood。生成阶段输出多组候选 pair，正面和背面一起展示，而不是孤立图片。用户选择成为下一条信号。

这就是偏好开始可操作的时刻。被选中的候选比 prompt 更有信息量，因为它是在备选项下做出的选择。被拒绝的候选也有价值，只要 rejection 绑定原因：颜色、位置、正反面不匹配、细节密度，或者某个 motif 不对。重点不是把反馈存成评论日志，而是把反馈转成下一组候选的约束。

这个经验后来也影响我做其他 AI 系统的判断：反馈只有在改变系统未来行为时才有价值。一个 thumbs-up 存在 analytics 里很弱；一个选择如果能改变 prompt control、style consistency、LoRA data selection 和 candidate filtering，它才成为产品状态。

正反面一致性，是图像变成衣服的第一关

单张正面图可以隐藏不一致。正反面 pair 会把问题暴露出来。

对服装来说，front/back consistency 不是装饰性功能，而是生产视角下的第一道测试：这个设计描述的是同一件衣服，还是两张无关图片？正反面不需要镜像，但必须属于同一套设计语言。颜色、motif、密度、位置和语义意图都应该连贯。一件黑色 T-shirt 正面是金色抽象波纹，背面可以更安静地延续它，但不能突然变成另一件完全不同主题的衣服。

1shanai 使用 FLUX LoRA 作为生成栈的一部分，但项目价值不是“用了 FLUX LoRA”这句话。真正重要的是围绕生成模型建立偏好和一致性流程。系统要保持同一设计意图跨视图存在，生成 candidate pairs，并让用户选择影响下一轮。LoRA 帮助风格收敛，产品状态帮助偏好收敛。

这个区分很重要。模型能力本身不能解决 garment coherence。生产交接需要稳定的正反面、被选择的版本、placement constraints，以及让打样伙伴或内部 operator 能理解“应该做什么”的连续信息。

正反面 pair 是一个很小的交付物，但它承担很大责任。它把用户 taste 从模型带进生产流程。

打样让质量从视觉问题变成物理问题

sampling 改变了质量的含义。

打样前，质量可以主要是视觉吸引力：生成图好不好看？打样后，质量变得更物理：印花颜色是否接近预期，图案位置在布料上是否平衡，正反面关系在真实尺寸下是否成立，用户收到实物后是否仍然喜欢。

很多 AI product demo 停在截图，因为物理世界反馈更慢。1shanai 在小规模上做了更慢的事。团队从 100+ beta 用户反馈中推进出约 40-50 件实体衣服，足够看到实物闭环提供的信息和图像选择不同。

样衣既是交付物，也是 evaluator。它测试生成 flat 里的假设：

假设样衣实际测试什么
图案适合衣服布料上的尺寸、位置和平衡
配色可接受印花和材质效果，而不是屏幕颜色
正反面是一件衣服两个视图是否读作同一个产品
用户选择代表真实偏好实物化之后是否仍然喜欢
交接信息足够清楚生产能否理解被选中的设计

这张表不是说团队解决了所有服装质量问题，而是说明有效评测面在哪里。实体衣服把生成设计变成了可以检验的对象。

下单交接是 AI 产品最容易漏掉的翻译层

从生成到订单的 handoff 容易被低估，因为它看起来像运营细节。实际上，它决定 AI 产品是变得有用，还是把责任转移给人。

生成系统输出候选图。订单 workflow 需要被选中的设计、正反面视图、garment type、colorway、尺码或 fit 假设、印花位置、revision notes、sampling status 和 delivery state。有人必须在这两个世界之间翻译。如果产品不定义这层翻译，operator 就只能从截图和聊天记录里重建意图。

1shanai 的 workflow 把这层翻译显性化：偏好输入变成 prompt 和 control generation；FLUX LoRA 产出候选；产品以正反面 flats 的形式展示 candidate pairs；用户选择收窄方向；sampling 产生第一轮物理检查；delivery 用真实物品关闭闭环。

因此，“AI apparel”不是一个单点能力，而是一串交付物：

阶段交付物
偏好输入风格意图、颜色、图形、fit、细节
生成控制prompt controls、LoRA-guided style direction、placement constraints
候选评审多组正反面 pair
用户选择被选版本、拒绝原因、refinement notes
打样交接面向生产的正反面设计和订单上下文
实物反馈样衣质量、用户反应、下一轮约束

当这些交付物显性存在，团队才有办法改进闭环。当它们隐含在聊天和截图里，系统就会退化成带人工履约的图库。

现有 beta 证据支持什么

1shanai 的证据边界很克制，也因此更可信。现有材料支持以下说法：

• 1shanai 是一个从广州出发的 AI-generated apparel 项目。

• 我以 co-founder 身份带过 7 人暑期团队。

• 项目从 T-shirt 开始，连接用户反馈、设计生成、正反面 flats、ordering、sampling 和 delivery。

• workflow 使用 FLUX LoRA，并围绕 front/back consistency 建立数据和流程。

• 用户的接受、拒绝和修改反馈会影响 prompt control、LoRA data selection、style consistency 和 candidate filtering。

• 项目积累了 100+ beta 用户反馈信号，并产出约 40-50 件实体衣服。

这些证据支持的是产品闭环主张：团队小规模测试了生成服装能否穿过偏好、正反面确认、打样和交付。它不支持大规模制造、稳定营收、成熟供应链、完整服装平台或严格学术化偏好评测结果。

这个边界让项目更有说服力。40-50 件衣服的意义不是销售规模，而是 closed loop 穿过了 screen-to-object boundary，暴露出纯图像 demo 看不到的问题。

如果继续做，应该评测什么

如果把 1shanai 继续推进成更严格的产品评测，我会先量化四件事。

第一是 selection quality。候选集中，用户有多大概率找到值得 refine 或 sampling 的设计？这比统计生成图片数量更有意义，因为系统的任务是降低选择负担，不是增加输出体积。

第二是 front/back coherence。用户和 operator 是否认为正反面属于同一件衣服？指标应该拆开看 semantic coherence、style coherence、color coherence 和 placement coherence。颜色一致不等于设计意图一致。

第三是 sample fidelity。实体衣服和确认过的 front/back flat 有多接近？这里要看印花位置、颜色、细节保留，以及用户收到样衣后的接受度。

第四是 feedback efficiency。一轮用户反馈之后，下一组候选是否更接近用户偏好？这是最难也最重要的指标，因为它测试反馈是否真的变成系统行为。

这些指标会让产品 insight 可检验，也会让项目保持诚实。一个模型即使能生成很多好看的衣服，只要用户不能选择、交接、打样和 refine，它在产品层面仍然不够好。

更大的经验

1shanai 训练出一个后来很有用的判断：AI 产品最难的部分，常常不是生成输出，而是把输出变成可评审、可交付、可反馈的行动。

在服装里，这个行动是打样和交付。在法律或税务工作里，它可能是带引用的草稿、案件交付物，或需要 approval gate 的邮件。两种场景都不只要求模型产出 plausible content。用户需要系统保留意图，暴露正确的中间交付物，支持人工审查，并让反馈改变下一轮行为。

对服装来说，生成图只是起点。真正的产品工作，从图像必须成为衣服的那一刻才开始。
